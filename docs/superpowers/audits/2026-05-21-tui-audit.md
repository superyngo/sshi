## Findings

### MINOR Inconsistent bounds checking in breadcrumb render
**File:** `src/tui/tabs/config_tab.rs:506,525,544`
**Symptom:** The `breadcrumb()` method fetches an entry's display name via
`config.host.get(*i).map(|h| h.name.as_str()).unwrap_or("?")` (safe), but
immediately after, in the `FieldTable` branch, passes `&config.host[*i]`
(direct indexing) to `host_descriptors()` / `check_descriptors()` /
`sync_descriptors()`. Same pattern for Check and Sync. If `i` were ever
stale (items vec not rebuilt after a config vec-length change), the name
would degrade gracefully to `"?"` while the descriptor call would panic.
**Fix sketch:** Replace `&config.host[*i]` with `config.host.get(*i)` and
early-return or use the `"?"` fallback consistently. ~6 lines changed.
**Disposition:** deferred-backlog

## Reviewed but not recorded

- **`len().max(1)` pattern:** 0 occurrences found. Task 2 cleanup confirmed.
- **59 `.unwrap()` in `src/tui/`:** ~40 in test code. Remaining 19 are in
  `config_tab` key handlers (`as_ref().unwrap()` / `as_mut().unwrap()` on
  `entry_form`, `confirm`, `direct_vec_editor`, `direct_group_picker`).
  All are locally provable: each function is called after a guard check
  verifies the Option is Some. Not a runtime risk with current state
  machine, but brittle to future changes.
- **4 `Mutex::lock().unwrap()` in `log_layer.rs`:** Standard pattern. The
  mutex is held briefly for `push`/`snapshot`/`len`/`is_empty`, none of
  which can panic. Poisoning would require a panic while holding the lock,
  which can't happen with the current code.
- **1 `expect()` in non-test code** (`app.rs:363`): `event_rx.take().expect(...)`
  in `run()`. Construction invariant — `event_rx` is always `Some` after
  `App` construction. Safe.
- **1 `panic!` in `config_tab.rs:3094`:** In test code only.
- **Hand-slice at `app.rs:2348`:** `self.checkout_snapshots[start..end]`.
  `set_dims` is called with `checkout_snapshots.len()` immediately before
  the slice, so `visible_range()` returns in-bounds values. Safe.
- **Hand-slice at `operate_tab.rs:610`:** `host_outcomes[start..(start+take).min(total)]`.
  `start` is clamped to `auto_start = total.saturating_sub(12)` and end is
  capped by `.min(total)`. Safe.
- **`adhoc_list_rows.max(1)` in `operate_tab.rs:299`:** Layout constraint
  (reserves minimum 1 row). Not the `len().max(1)` pagination anti-pattern.
- **`filter_popup.take().unwrap()` at `app.rs:973`:** Called inside a match
  arm where the popup was just confirmed as Some. Safe.
- **Signal handling:** Properly implemented via async `mpsc` channel.
  `spawn_signal_listener` handles SIGHUP/SIGTERM/SIGINT on Unix, ctrl_c on
  Windows. Main loop drains non-blocking. Clean shutdown path saves state.
- **State persistence (`persist.rs`):** Graceful degradation on load failure
  (returns defaults, never panics). Atomic write via `NamedTempFile::persist`.
  `save()` returns `Result<()>` with no panics in non-test code.
- **`Viewport` component:** Robust — `visible_slice()` clamps both start and
  end against actual slice length, handling stale `set_dims` calls safely.
- **File sizes:** `config_tab.rs` at 3436 lines and `app.rs` at 2638 lines
  are large monoliths. Not a correctness issue; noted for future refactoring
  consideration.
